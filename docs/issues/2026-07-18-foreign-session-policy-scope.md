# 外部会话策略错误扩散到无关资源

## 问题摘要

Workloop 当前可能根据 Hook 的启动工作区和该工作区的会话绑定，对整个工具调用应用外部会话限制，而没有先解析操作实际影响的工作区和资源。

这会导致一个活动任务的保护范围从它实际占用的资源扩散到：

- 其他独立仓库中的操作；
- 同一仓库内与该任务不冲突的文件、目录或其他资源；
- 仅包含相关命令文本、但不会执行该命令的只读工具参数。

`git -C` 只是其中一个复现形式，不是问题边界。根本问题是策略作用域以 Hook 启动位置为中心，而不是以操作的实际影响范围为中心。

## 实际行为

仓库 A 中存在一个由会话 A 绑定的活动 Workloop 任务。会话 B 调用工具时，进程从仓库 A 启动，因此被识别为外部会话。

即使操作明确指向仓库 B，或者只影响仓库 A 中不属于任务 A 资源范围的内容，Workloop 仍可能应用任务 A 的外部会话限制并拒绝操作。

跨仓库示例：

```sh
git -C <仓库B> status
cd <仓库B> && npm test
npm --prefix <仓库B> run build
cargo test --manifest-path <仓库B>/Cargo.toml
rm <仓库B>/generated-file
cp <源文件> <仓库B>/destination
command > <仓库B>/output.txt
```

同一仓库内的示例：

```text
任务 A 的资源范围：
- src/auth/**
- tests/auth/**

任务 B 的资源范围：
- src/billing/**
- tests/billing/**
```

两个任务位于同一个仓库，但资源范围没有交集，不应因为任务 A 处于活动状态就阻止任务 B。

## 预期行为

Workloop 应根据操作实际影响的资源确定策略作用域，而不是仅根据 Hook 从哪个目录启动来判断。

仓库 A 的会话绑定应该保护：

- 任务声明的文件、目录、生成物和其他资源范围；
- 仓库 A 的 Workloop 与 Git 控制状态；
- 无法解析出其他目标、并继承仓库 A 作为作用域的操作；
- 与活动任务资源范围发生实际交集的操作。

它不应该自动管理：

- 目标完全位于独立仓库 B 的操作；
- 同一仓库内与活动任务资源范围不相交的操作；
- 不会产生写入或外部副作用的只读操作。

目标属于仓库 B 时，应根据仓库 B 自身的 Workloop 状态独立判断。目标位于同一仓库但不与任务资源相交时，应允许并行，或者按未跟踪工作策略处理，而不是锁定整个仓库。

## 正确的决策模型

当前行为近似于：

```text
Hook 启动工作区
→ 查找活动任务
→ 判断会话所有权
→ 对整个操作应用策略
```

期望行为是：

```text
操作
→ 解析实际影响和副作用
→ 解析目标资源及所属工作区
→ 与各工作区的活动任务资源范围比较
→ 只对发生交集的部分应用相应会话策略
```

策略判断的核心不应是“是否在同一个仓库”，而应是“操作是否会影响受保护资源”。

## 作用域解析要求

在应用外部会话策略之前，Workloop 应解析：

- 工具调用中显式指定的工作目录；
- `cd` 等 Shell 目录切换；
- 绝对路径和相对路径形式的输入、输出及修改目标；
- 输入和输出重定向；
- `-C`、`--prefix`、`--cwd`、`--manifest-path` 等命令专用工作区参数；
- 传给编辑、补丁、删除、复制和移动工具的文件路径；
- 通过 Shell、环境包装器或进程启动器执行的嵌套命令；
- 构建工具、包管理器和测试工具隐式影响的工作区；
- 同时跨越多个文件、目录或工作区的操作；
- 没有文件目标但存在发布、网络、进程或其他外部副作用的操作。

## 同一仓库内的资源级并行

同一个仓库存在活动任务，不等于整个仓库都应该被独占。

Workloop 应至少支持文件和目录级资源范围：

| 场景 | 预期处理 |
| --- | --- |
| 不同仓库 | 根据各自的 Workloop 状态独立判断 |
| 同一仓库、不同且不重叠的文件 | 允许并行 |
| 同一仓库、不同且不重叠的目录 | 允许并行 |
| 同一仓库、只读操作 | 不应被写入锁阻止 |
| 同一仓库、资源范围发生交集 | 阻止、要求加入任务，或要求拆分操作 |
| 修改 Workloop 或 Git 控制状态 | 始终作为受保护资源处理 |
| 无法解析实际目标 | 返回明确的作用域无法确定错误 |

### 同一文件的情况

两个任务修改同一个文件时，是否冲突取决于 Workloop 能够证明的资源粒度。

在只有文件路径资源模型的情况下，同一文件应默认视为冲突，因为格式化、import 排序、代码移动、重命名和基于旧版本生成补丁都可能让看似独立的修改互相覆盖。

如果未来支持“同一文件、不同区域”并行，需要额外记录并验证：

- 文件的基础版本或内容摘要；
- 占用的符号、语法节点、代码区域或行范围；
- 实际补丁影响范围；
- 格式化或代码生成是否会重写整个文件；
- 每次写入后，两个任务的影响范围是否仍然不相交。

只有能够持续证明这些范围不冲突时，才应允许同一文件并行修改；无法证明时应保守地判定为冲突。

## 混合作用域操作

如果同一个操作同时影响任务受保护资源和其他资源：

- 与活动任务资源范围相交的部分必须继续受到保护；
- 不相交的部分不应被错误归属给该任务；
- 如果 Workloop 无法安全分离目标，应要求拆分操作；
- 错误信息应指出具体冲突资源或无法解析的作用域，而不是笼统提示整个仓库被另一个会话占用。

## 全局操作

没有明确文件系统或仓库目标的操作，仍然可以受到独立的宿主级安全规则约束。

但是，除非当前工作区确实是该操作的实际作用域，否则不应把这种限制描述为当前工作区会话绑定产生的结果。宿主级风险限制与任务资源冲突应该使用不同的错误类型和提示。

## 根本原因

外部会话身份目前在操作目标和副作用完全解析之前，就根据 Hook 启动工作区确定。部分命令级拒绝也发生在资源归属判断之前。

因此，一个工作区的活动任务可能影响与它没有资源关系的操作。这既会阻止跨仓库工作，也会阻止同一仓库中的非冲突并行任务。

## 验收标准

- 只影响仓库 B 的操作不会被仓库 A 的活动任务阻止。
- 仓库 B 根据自身的 Workloop 状态独立判断。
- 同一仓库内，不同且不重叠的资源范围可以并行操作。
- 外部会话只被禁止修改与活动任务资源范围相交的目标。
- 只读操作不因活动任务的写入保护而被拒绝。
- 文件工具和 Shell 命令使用一致的资源解析与归属模型。
- 显式工作目录、目录切换、绝对路径、相对路径、重定向和命令专用工作区参数均有覆盖。
- 混合作用域操作能够保护冲突资源，同时不会错误接管非冲突资源。
- 同一文件在无法证明区域不冲突时默认判定为冲突。
- 如果支持同一文件不同区域并行，必须验证基础版本、补丁范围和格式化影响。
- 无法解析目标时，返回明确的作用域解析错误，而不是直接锁定整个仓库。
- 真正作用于活动任务资源范围的操作继续受到现有策略保护。
- 冲突提示列出具体资源、所属任务和推荐动作。
- 回归测试覆盖独立仓库、同一仓库的非重叠任务、关联工作树、嵌套命令、文件操作、构建工具和包管理器工作区参数。

## 非目标

- 不取消活动任务的资源保护。
- 不允许无法证明安全的同一文件并发写入。
- 不把宿主级安全规则降级为工作区级规则。
- 不要求 Workloop 自动合并存在冲突的修改。
