  | **语境独立性**（`fresh_context`） | **在边界上**——**必要条件双宿主对称可观测**（`agent_id`，票 12 实测 + Codex 源码级对称），**属性本身不可观测**（P2/P3：橡皮图章 subagent 与诚实 subagent 机器不可判定）。**已由票 12 裁完，不再是雾。** |
  | **模型多样性**（`second_model`） | **不对称**——Codex 的 hook payload 有 `model` 字段，**Claude Code 的没有**（票 12 实测 keys 无 model）→ **内核押不了它**，否则即 host-dependent。 |

  **票 12 部分证否了本条原来的话**：本条原断言「这条信任链缺一个 runtime 能独立观测的锚点」，08 据此认定评审面**有锚**（`review.acting_session`）而 graduate 了它。**两边都不全对**——`acting_session` 在 CLI 路径上**不是观测**（可伪造），真锚是**钩子 payload 里的 `agent_id`**，而它只锚得到必要条件。

  **graduate 条件（按面分列）**：**人**——出现 `user` 授权被误用/误记的样本（`agent_id` 路线已排除，别再指望它）。**2026-07-16 票 14 给人-面挂上又一个等着的消费者**：人-面身份锚若出现，封条 acceptance 的 `user` 署名可验证、typed `human_verification` 一等化可重问（见[封条形态决策记录](../../docs/decisions/2026-07-16-unreachable-business-truth-seal.md)翻盘条件）；**模型多样性**——Claude Code 的 hook payload 出现模型标识，届时该面变对称、可重问（**票 16 已给它一个等着的消费者**：`second_model` 的梯顶锚，届时可补验证——见决议 5）。**2026-07-16 票 16/17 给人-面开了一条新路，17 已回**：宿主权限档确实进 payload（`permission_mode`，与 `agent_id` 同级：宿主授权、agent 写不到），**人-面在「权力钥匙」子问题上就此获得第一个、也是目前唯一的机器锚**——但只有 `bypassPermissions` 这一颗双宿主对称（细档位在 Codex 侧被运行时折叠成错值；sandbox 轴两边都不进 payload）。它锚不到 `granted_by` 的真伪，锚得到「宿主放行时钥匙是否整体预转开」。

- **写入量／改动量该不该作为第二个规模信号，以及 `many_touched_files` 的 >10 阈值**（票 11 决议 1/2 留下）——**今天结构上不可裁**：`many_touched_files` 在 10/10 任务里不可达（true 分支从没执行过一次），账本里也没有 `touched_files` 的分布可比，故「>10 对不对」**不可证伪**，而「换成 writes 更好吗」是在一具尸体身上争穿什么衣服。第一性上写入量**合格**（观测、对撒谎免疫），只是「独立声称数」的**更差代理**（文件数 > 改动量 > 写入次数）——故未裁死，只是等证据。**graduate 条件**：11 的守卫修复落地 **且** v3 发布（fold 能给出 `touched_files` 分布）之后，`many_touched_files` 第一次在真实任务上出现在 reasons 里——届时才同时具备「阈值校准所需的分布」与「规则真会开火的证据」。**注意**：它的不可达根因是 07 已裁的饱和链（command 判据 → 必有 proof gap → 地板必 ≥ substantial），故它**很可能随票 10 决议 4（`kind:"file"` + `.mjs`）第一次上岗**——repo 桶被进入 → 判据够得着 `adequate` → 地板才可能停在 routine。详见[风险地板口径决策记录](../../docs/decisions/2026-07-16-risk-floor-calibration.md)。**2026-07-16 票 16 更新**：不可达的**结构根因已根裁**——它死于两本账被 `max()` 焊死（缺口轴恒先饱和、赌注轴守卫永不被求值）；两账拆分后赌注轴独立结算，规则必然开始被求值，且「实得赌注」把 touched_files 升为关单地板的一等输入。**阈值校准仍等 v3 分布，graduate 条件不变**。

- **无机器判据的仓内工作（规划/裁决/文书）怎么接进 taskloop**——**注意与 [票 14](issues/14-unreachable-business-truth-seal.md) 分辨**（2026-07-16 由票 10 出票）：本条是**没有客观真相**的活（规划/裁决/文书）；14 是**有客观真相但机器够不到**的活（跨系统业务）。两者都落不进 workloop，但病因不同——本条是判据不存在，14 是判据只够得着真相的影子。2026-07-15 现场证据：本图 03 session 在 `.scratch/` 里改地图，被 runtime 的 untracked gate `deny`（第 2 个仓内文件），owner 当场移除 hook 才得以收尾。这类活既不是 workloop（无可复跑判据，硬造就是「为过闸门造传感器」）、也不完全是 judgmentloop；而票 05 决议 1「动手前开单为纪律」在它身上落不了地。**07 已裁，并未救它**（此前本条指望 07）：07 裁的是判据**弱**的时候怎么办（作者+时序分级、开单自带即干净），而这类活**连判据都没有**——不是判据弱，是判据不存在。07 决议 4「开单自带 = 干净，不论谁写」反而把这个缺口照得更亮：**它奖励「动手前立字据」，可这类活立不出机器能读的字据。****闸门是对的，纪律是对的，但这类工作没有合法形态**——这是维度 1／10 的接缝。等 05 的纪律真正跑起来后再 graduate（届时会有更多同类样本）。**2026-07-16 票 13 给它装上了传感器规格**（证据书 + nudge 档全档位记录）：它等的「同类样本」将以 untracked 观测行的形式积累，前提是 #04 的模式梯执行项落地。
